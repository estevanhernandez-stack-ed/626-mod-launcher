# Launcher 0.8.1 — plugin delivery UX fixes (post-live-smoke) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Per-task implement → review → fix gated, then a whole-branch review. Steps use `- [ ]`.

**Goal:** Close the three gaps the 2026-06-19 live smoke of v0.8.0 exposed in the *consumer* side of plugin delivery, so the off-Store Nexus plugin reaches already-connected users, lights the UI without a restart, and is visibly controllable. Launcher-only (FULL flavor); no `626-mod-plugins` changes.

**Background (what the smoke showed):** v0.8.0 published + the signed Nexus plugin published + cryptographically verified. On the rig: the connect-time fetch *did* run, downloaded + verified + installed `nexus.dll`, wrote `installed-plugins.json` — but (a) it only fires on a fresh `ConnectAsync`, so an already-connected user (key persisted) never triggered it; (b) the mid-session hot-load did not light the Nexus surfaces — a full restart was needed; (c) the whole path is fail-silent, so the user couldn't tell what happened. A stale hand-dropped `ModManager.Plugin.Nexus.dll` from B1 dev-testing also sat in the plugins dir next to the feed-installed `nexus.dll`.

**Tech Stack:** .NET 10 / WinUI 3. All changes App-side, `#if FULL` where they touch the plugin host/feed.

## Task 1: PluginFeedSource returns an outcome (refactor) + manual entry point

**Files:** `src/ModManager.App/Services/PluginFeedSource.cs`

**Interfaces:**
- Produces: `enum PluginFetchOutcome { NotApplicable, UpToDate, Installed, Failed }` and a result record `sealed record PluginFetchResult(PluginFetchOutcome Outcome, string? Version, string? Message)`.
- `public async Task<PluginFetchResult> FetchAsync(bool force)` — the core fetch (today's `MaybeFetchOnConnectAsync` body) made to RETURN a result instead of pure fail-silent. `force` bypasses the debounce + the `KeepPluginsUpdated` gate (the manual button + first-install path). Still never throws (catches → `Failed` with a friendly message).
- `MaybeFetchOnConnectAsync()` stays (connect-trigger, fire-and-forget) but delegates to `FetchAsync(force: !anyInstalled)` and keeps raising `PluginLoaded` on a successful load.

- [ ] **Step 1:** Extract the existing body of `MaybeFetchOnConnectAsync` into `FetchAsync(bool force)` that returns `PluginFetchResult`:
  - keep the `_running` Interlocked guard (concurrent calls → `NotApplicable` with "already running").
  - `NotConnected`/no-domain → `NotApplicable`.
  - if `!force` and a plugin is installed: honor `KeepPluginsUpdated` + the 24h `ShouldPoll` debounce (→ `NotApplicable`/`UpToDate` when skipped); `force` bypasses both.
  - run `PluginFeedInstaller.RunAsync`; map: installed something → hot-load each via `PluginHost.LoadOne`, raise `PluginLoaded` if any loaded, return `Installed` with the version; installed nothing but the feed was reachable → `UpToDate`; the download delegate returned null (offline/404) or an exception was caught → `Failed` with a short reason ("couldn't reach the plugin feed", "feed returned nothing", etc.).
  - stamp on a real fetch attempt only (the existing finally-stamp logic, preserved — a `NotApplicable` debounced skip must not stamp).
- [ ] **Step 2:** `MaybeFetchOnConnectAsync` becomes `=> _ = FetchAsync(force: InstalledPluginsStore.Read(RecordPath).Count == 0)` (fire-and-forget; first-install forces, otherwise debounced). Verify the connect path still compiles + the FULL build is 0 errors.
- [ ] **Step 3:** Commit (`refactor(plugins): PluginFeedSource.FetchAsync returns an outcome (manual + connect share it)`).

> No unit test added here (App `#if FULL`, network glue); the install/verify/gate logic is already covered by `PluginFeedInstaller` Core tests. Build-verified.

## Task 3: Mid-session hot-load lights the Nexus UI (no restart)

**Files:** `src/ModManager.App/ViewModels/MainViewModel.cs` (`WirePluginFeed`, ~line 1367)

**Root cause (confirmed):** `ReloadModsAsync` (line 359) re-lists from persisted `metadata.json` via `ModListing.Resolve` — it does NOT re-run Nexus identify or the endorse/stats sync. So when the plugin hot-loads mid-session, a plain `ReloadModsAsync` surfaces only already-persisted Nexus metadata (none, for mods identified before the plugin existed) → no hearts. A restart worked because the game-load path runs identify/poll *with* the plugin already registered.

- [ ] **Step 1:** Change the `WirePluginFeed` `PluginLoaded` handler so that, on the UI thread (`_dispatcherQueue.TryEnqueue`), instead of just `ReloadModsAsync()` it runs the **full re-detect/enrichment path** that the game-load uses — `RedetectActiveAsync()` (line 2104) — which re-runs the scan + Nexus identify + the stats/endorse sync with the now-registered source, then rebuilds the rows. Confirm `RedetectActiveAsync` triggers the Nexus identify + endorse enrichment (it's the "Redetect" button's path); if it does not itself sync endorse state, also invoke the same game-load Nexus enrichment (`NexusUpdatePoll`/`RefreshNexusStatsAsync`-equivalent) so hearts fill. Keep the `OnPropertyChanged(nameof(NexusActionsAvailable))` + `NexusActionsVisibility` notifications (so the "Refresh Nexus stats" item appears).
- [ ] **Step 2:** Build FULL. Reason through the flow in the report: hot-load → `PluginLoaded` → dispatcher → re-detect (identify with source) → rows rebuilt with Nexus enrichment → hearts + actions visible, no restart.
- [ ] **Step 3:** Commit (`fix(plugins): hot-load re-identifies so Nexus surfaces light up without a restart`).

> If `RedetectActiveAsync` is heavier than wanted on every hot-load, scope it: only re-detect when the load actually added a new source (the handler already fires only on `PluginLoaded`). Build-verified + live-smoked.

## Task 5: Loader ignores non-canonical leftover plugin dlls

**Files:** `src/ModManager.App/Services/PluginHost.cs` (`LoadAll`)

**Why:** `LoadAll` enumerates every `*.dll`. A stale hand-dropped `ModManager.Plugin.Nexus.dll` (B1 dev-testing) sat beside the feed-installed `nexus.dll`; alphabetically it loads first and, if it verified, would register `nexus` and shadow the real one (the feed always writes the id-named `nexus.dll`). Normal users never hand-drop, but harden so a leftover can't shadow the feed plugin.

- [ ] **Step 1:** In `LoadAll`, load only dlls whose name matches the feed's id-naming (`<id>.dll`, i.e. lowercase, no `.` beyond the extension) OR, simpler + sufficient: prefer the `installed-plugins.json` record — load exactly the recorded `<id>.dll` files, ignoring any other dll in the dir. Keep `LoadOne` unchanged (still verifies the sig). A dll present-but-not-recorded is skipped (logged at debug). This keeps the fail-closed signature gate AND stops a stray verified-but-stale dll from shadowing.
- [ ] **Step 2:** Build FULL + STORE (STORE: PluginHost is `#if FULL`, unaffected). Core suite green (PluginHost isn't in Core; CorePurity unaffected).
- [ ] **Step 3:** Commit (`fix(plugins): load only feed-recorded plugin dlls, ignore stale leftovers`).

> Note for the smoke doc: existing rigs may have the stale `ModManager.Plugin.Nexus.dll` + `.sig` — harmless after this change (ignored), but a one-line cleanup note is worth adding to `docs/smoke-tests/pending.md`.
