# Public "supported games" surface — design

**Date:** 2026-07-03
**Status:** Spec (approved in-conversation). Implementation lives entirely in the **626-game-manifest** repo; this doc is the system-of-record copy. Small feature — one generator script + CI wiring + README badge.

## The problem

The feed repo publishes a signed `games-manifest.json` (143 entries: 103 engine-curated, 40 nexus-only, 18 featured) that only the launcher reads. There's no human-readable "what games are supported" page on GitHub, and no stable machine-readable surface for other consumers — the 626 Labs hub website and the Discord server/bot Este is scaffolding.

## Approved decisions

1. **Same publish rail as the manifest:** CI generates the public files and commits them to `main` alongside `games-manifest.json` — consumers fetch stable `raw.githubusercontent.com` URLs (CORS-open, so the hub fetches client-side). No GitHub Pages, no release assets, no new infra.
2. **JSON only for Discord** — the bot reads `supported-games.json` on demand; no CI announce webhook (announcements are the bot's job later if wanted).
3. **The generator lives in the feed repo, not ManifestMiner** — the miner ships facts (launcher repo); presentation belongs to the feed repo so page tweaks never touch the launcher. Este: "we handle the lifting where we have the tools."

## Outputs (three files, committed by CI)

### 1. `SUPPORTED-GAMES.md` — the human page
- Header: total / engine-curated / nexus-only counts + generated date (UTC).
- **Featured** section: the quick-pick games in `featured` rank order.
- **Engine-curated** table (sorted by name): Name · Engine · Mod path · Steam link · Nexus link.
- **Nexus-only** table (sorted by name): Name · Steam link · Nexus link.
- One-line tier explainers (engine-curated = quick-pick setup, the launcher knows the engine + mod folder; nexus-only = identified on Nexus, engine folder-detected at runtime) + a "want a game added?" link to the game-request issue template.
- Footer: "Generated by CI from games-manifest.json — do not edit by hand."

### 2. `supported-games.json` — the consumer contract
```json
{
  "schemaVersion": 1,
  "generatedUtc": "2026-07-03T05:12:00Z",
  "counts": { "total": 143, "engineCurated": 103, "nexusOnly": 40 },
  "games": [
    {
      "id": "elden-ring",
      "name": "ELDEN RING",
      "steamAppId": "1245620",
      "tier": "engine-curated",
      "engine": "fromsoft",
      "featured": 3,
      "steamUrl": "https://store.steampowered.com/app/1245620/",
      "nexusUrl": "https://www.nexusmods.com/eldenring"
    }
  ]
}
```
- camelCase (house convention). `engine`/`featured`/`nexusUrl`/`steamUrl` are **omitted** (not null) when not applicable — consumers check for presence.
- `tier` derived exactly as the launcher facades do: engine present → `engine-curated`; else nexusDomain present → `nexus-only`. (The published manifest contains only entries that earned a tag, so nothing else appears.)
- **Stability guarantees:** stable raw URL on `main`; `schemaVersion` bumps only on breaking shape change; additive fields allowed without a bump.

### 3. `badge.json` — shields.io endpoint schema
```json
{ "schemaVersion": 1, "label": "supported games", "message": "143", "color": "#17d4fa" }
```
Rendered anywhere via `https://img.shields.io/endpoint?url=<raw badge.json url>` — README + hub both use it.

## The generator — `tools/generate-public.py` (feed repo)

- Python 3 stdlib only (json/datetime/argparse; ubuntu-latest has it). No dependencies, no install step.
- Input: path to the freshly built `games-manifest.json` (+ `--out-dir`, default repo root). Timestamp from `$SOURCE_DATE` env or `--generated-utc` arg (CI passes it; no hidden clock reads for testability).
- Reads the manifest's camelCase shape (`games[]` with `id`, `name`, `engine`, `stores.steamAppId`, `nexusDomain`, `modPath`, `featured`), derives tier + URLs (`store.steampowered.com/app/<id>/`, `nexusmods.com/<domain>`), emits all three files deterministically (sorted, stable ordering → clean diffs).
- **Self-test mode:** `--self-test` runs embedded fixture assertions (3-game fixture: one engine-curated + featured, one nexus-only, one with no steam id) — asserts tier split, counts, badge message, markdown contains/omits the right rows, URLs correct. Exits non-zero on failure.

## CI wiring (`build-manifest.yml`)

After the mine+sign step, before the publish step:
1. `python3 tools/generate-public.py --self-test` (gate — a generator bug fails the build before anything publishes).
2. `python3 tools/generate-public.py launcher/tools/ManifestMiner/out/games-manifest.json --generated-utc "$(date -u +%Y-%m-%dT%H:%M:%SZ)"`.
3. The publish step's `git add` grows to include `SUPPORTED-GAMES.md supported-games.json badge.json` — same commit as the manifest, so the public files can never drift from the signed data. The existing no-change guard covers all of it.

## README wiring (feed repo)

Badge at the top (`img.shields.io/endpoint?url=…badge.json`) + one line: "**See the full [supported games list](SUPPORTED-GAMES.md)** — regenerated automatically as curation lands."

## Consumers

- **Hub website:** fetch `supported-games.json` raw URL client-side (CORS-open). Cover art derivable from `steamAppId` via Steam's public CDN (`cdn.cloudflare.steamstatic.com/steam/apps/<id>/library_600x900.jpg`, with the hashed `store_item_assets` fallback for brand-new games — see the launcher's `CoverCache` two-tier pattern).
- **Discord bot:** fetch the same URL on demand (e.g. a `/games` command). No CI-side coupling.
- **Launcher:** unaffected — the signed `games-manifest.json` contract is untouched; this is a read-only projection born in the same CI commit.

## Testing

Self-test fixture assertions run in CI before every publish (see generator). The live run then processes the real manifest; the same-commit rule makes drift structurally impossible. No launcher-repo tests needed (nothing in the launcher changes).

## Non-goals

- No GitHub Pages site (raw + GitHub-rendered markdown suffice).
- No cover images embedded in the JSON (consumers derive from `steamAppId`).
- No Discord announce webhook (bot's job).
- No change to the signed manifest, its schema, or the launcher.
