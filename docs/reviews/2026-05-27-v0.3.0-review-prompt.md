You're doing a critical-eye code review of the 626 Mod Launcher at tag v0.3.0. Pretend you've never seen this codebase before — your job is to find what the author missed.

## Context

- This is a .NET 10 / WinUI 3 mod manager for FromSoft + UE games + others. It's the native rewrite of an earlier Electron app.
- Two operating laws govern every file op:
  1. Reversibility — disabling moves files, never deletes; replacing snapshots originals first
  2. Pure-core / thin-shell — Core has no Electron/WinRT; App is a thin IPC shell
- See CLAUDE.md, README.md, and `references/engines.md` for the broader rules.

## What landed in v0.3.0

## What I want from you

Focus, in this order:

1. **Safety / blast radius** — every F2 + F3 install path touches the game root or game data. Did the implementer miss any case where a malformed catalog entry or zip could damage a user's install? Look hard at `FrameworkInstaller.Install` (validate-then-extract) and the forbidden-paths gate. Look at `DirectInjectConfigOverrides.Save` — should it validate the override path before writing?
2. **Reversibility holes** — every file op is supposed to be undoable. Trace what happens when a framework install fails mid-extract. Trace what happens when a user manually edits `config-overrides.json` to point at a missing file (does the resolver handle it gracefully?).
3. **Pure-core hygiene** — Core has zero WinUI / Electron / WinRT references. Verify no leakage in the new Core files.
4. **camelCase JSON convention** — all on-disk JSON written by the launcher must be camelCase (Electron-shared convention). Verify `FrameworkInstallManifest`, `DirectInjectConfigOverrides`, and any other new on-disk shape comply.
5. **Test gaps** — what real-world scenarios aren't covered? Forbidden-path gate test cases, partial-extract recovery, malformed JSON inputs, etc.
6. **Naming / API smells** — `KnownDirectInjectMod` is the first "unified catalog" type; the spec says future phases fold Tools and Frameworks in. Does the kind-tagged shape actually support that, or is the schema too narrow?
7. **UX gaps** — drop-zip flows have confirmation dialogs; INI editor has Restore Previous; Settings has Uninstall + Override. What's missing from the user's recovery path? What error state will leave a user stuck?
8. **Honor-the-builders** — the launcher recognizes third-party tools/frameworks but never bundles them. Verify NOTICE attribution is complete + the "metadata only, never bundled" language is present wherever applicable.

## What's explicitly out of scope

- The Saves dialog's snapshot/restore engine (no v0.3.0 changes there)
- The mod scanner + variant grouping (no v0.3.0 changes)
- The Vortex / MO2 coordination law (untouched)

- A summary at the top: was this batch of work shipped at acceptable quality, or are there must-fix bugs before another release?

Use the existing `docs/reviews/` files as templates for tone + severity convention.

Read CLAUDE.md before you start so you understand the operating laws this work is supposed to obey.
